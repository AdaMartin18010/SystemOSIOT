## 1. DNS 基础与威胁模型

DNS（Domain Name System）将人类可读的域名解析为 IP 地址，核心规范见 [RFC 1034](https://datatracker.ietf.org/doc/html/rfc1034) 与 [RFC 1035](https://datatracker.ietf.org/doc/html/rfc1035)。由于其最初设计未考虑安全，传统 DNS 面临以下威胁：

| 威胁 | 说明 | 典型攻击 |
|---|---|---|
| 缓存投毒 | 向解析器注入伪造记录 | Kaminsky 攻击 |
| 窃听 | 明文 UDP/53 传输可被监听 | 运营商或本地网络监听 |
| 篡改 | 中间人修改响应 | 恶意热点、国家防火墙 |
| 拒绝服务 | 放大攻击或枚举查询 | DNS 放大 DDoS |

## 2. DNSSEC

DNSSEC（DNS Security Extensions）通过数字签名链为 DNS 数据提供来源验证与完整性保护，核心 RFC 为 [RFC 4033](https://datatracker.ietf.org/doc/html/rfc4033)、[RFC 4034](https://datatracker.ietf.org/doc/html/rfc4034)、[RFC 4035](https://datatracker.ietf.org/doc/html/rfc4035)。

### 2.1 关键记录类型

| 记录 | 作用 |
|---|---|
| DNSKEY | 区域公钥 |
| RRSIG | 资源记录集的数字签名 |
| DS | 父区域对子区域 DNSKEY 的哈希委托 |
| NSEC / NSEC3 | 认证不存在记录 |

### 2.2 信任链

```text
根区 DNSKEY
   |
   | 签名 . (顶级域) 的 DS/DNSKEY
   v
.com DNSKEY
   |
   | 签名 example.com 的 DS
   v
example.com DNSKEY
   |
   | 签名 www.example.com 的 A 记录
   v
www.example.com A + RRSIG
```

根区 DNSSEC 自 2010 年起由 ICANN 部署并管理 KSK（Key Signing Key）轮转。

来源：[RFC 4033-4035](https://datatracker.ietf.org/doc/html/rfc4033)、[RFC 6781](https://datatracker.ietf.org/doc/html/rfc6781)（DNSSEC 操作实践）、[RFC 8199](https://datatracker.ietf.org/doc/html/rfc8199)（DNS 术语更新）。

## 3. DNS over TLS (DoT)

DoT 在 TLS 之上封装 DNS 查询，使用 TCP/853 端口，定义于 [RFC 7858](https://datatracker.ietf.org/doc/html/rfc7858) 与 [RFC 8310](https://datatracker.ietf.org/doc/html/rfc8310)（TLS 配置要求）。

- 提供机密性与完整性，防止窃听与篡改。
- 独立端口 853 便于网络策略识别，但也更容易被封锁。

## 4. DNS over HTTPS (DoH)

DoH 将 DNS 查询作为 HTTPS 流量传输，定义于 [RFC 8484](https://datatracker.ietf.org/doc/html/rfc8484)。

- 复用 HTTPS（TCP/443 或 QUIC/443），与传统 Web 流量难以区分。
- 天然继承 HTTP/2 的多路复用与 TLS 1.3 的加密。
- 现代浏览器（Firefox、Chrome、Edge、Safari）与操作系统已广泛支持。

来源：[RFC 8484](https://datatracker.ietf.org/doc/html/rfc8484)。

## 5. DoH / DoT / DNSSEC 的关系

| 机制 | 保护层级 | 解决问题 |
|---|---|---|
| DNSSEC | 数据认证 | 防止缓存投毒、篡改 |
| DoT | 传输加密 | 防止窃听、中间人 |
| DoH | 传输加密 + Web 流量混淆 | 防止窃听、审查、中间人 |

DNSSEC 不加密查询内容，仅保证记录真实性；DoH/DoT 不验证 DNS 数据本身的真实性，需与 DNSSEC 结合才能实现端到端安全。

## 6. 部署现状与工程实践

- **DNSSEC**：根区、多数 TLD 与大型 CDN 已签署；递归解析器（BIND 9、Unbound、Knot Resolver）支持验证。
- **DoT**：Android 9+ 支持私有 DNS（DoT），Cloudflare、Quad9、Google 提供公共 DoT 服务。
- **DoH**：Cloudflare 1.1.1.1、Google 8.8.8.8、Quad9 9.9.9.9 提供公共 DoH；企业网络常通过 DNS 过滤代理管理。

## 7. 批判性分析

- **隐私悖论**：DoH 防止本地网络窃听，但将解析数据集中于少数公共解析器，带来新的隐私集中化风险。
- **网络可见性下降**：DoH 使企业/学校/家长控制型 DNS 过滤更困难，引发网络治理争议。
- **DNSSEC 部署复杂性**：密钥管理、KSK 轮转、NSEC3 枚举风险增加运营负担。
