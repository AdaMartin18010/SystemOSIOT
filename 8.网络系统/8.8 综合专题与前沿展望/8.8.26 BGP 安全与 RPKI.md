## 1. BGP 安全威胁模型

BGP-4（[RFC 4271](https://datatracker.ietf.org/doc/html/rfc4271)）基于路径向量与 TCP 会话信任，本身缺乏对路由前缀来源的密码学验证，导致两类主要威胁：

- **路由劫持（Route Hijack）**：自治系统（AS）宣告不属于其授权的前缀，吸引流量经过攻击者网络。
- **路由泄露（Route Leak）**：AS 违反出口策略，将学到的路由传播到不应接收的对等体，造成全局流量绕行或黑洞。

典型案例包括 2008 年巴基斯坦 Telecom 的 YouTube 前缀劫持、2018 年俄罗斯 Telecom 的 Google 路由泄露等。

来源：[RFC 4271](https://datatracker.ietf.org/doc/html/rfc4271)、[RFC 7908](https://datatracker.ietf.org/doc/html/rfc7908)（路由泄露定义）。

## 2. RPKI 与 ROA

资源公钥基础设施（Resource Public Key Infrastructure, RPKI）由 IETF SIDROPS 工作组定义，核心文件包括：

| 组件 | RFC | 作用 |
|---|---|---|
| RPKI 架构 | [RFC 6480](https://datatracker.ietf.org/doc/html/rfc6480) | 定义使用 X.509 证书将 IP/AS 资源与持有者绑定的框架 |
| ROA (Route Origin Authorization) | [RFC 6482](https://datatracker.ietf.org/doc/html/rfc6482) | 允许资源持有者声明哪些 AS 被授权起源哪些前缀 |
| RPKI 证书配置文件 | [RFC 6487](https://datatracker.ietf.org/doc/html/rfc6487) | 规定资源证书扩展与验证规则 |

ROA 三元组：

```text
{ prefix, maxLength, originAS }
```

- 资源持有者通过 RIR（Regional Internet Registry）托管的 RPKI 发布 ROA。
- 验证者下载并验证 ROA，用于后续路由过滤。

## 3. Route Origin Validation (ROV)

ROV 是网络运营商使用 RPKI 数据对 BGP 宣告进行来源验证的过程：

```text
收到 BGP UPDATE
  |
  v
提取 {prefix, originAS}
  |
  v
匹配 RPKI ROA
  |-- 有效（Valid）  -> 接受或提升优先级
  |-- 无效（Invalid）-> 丢弃或降级
  |-- 未找到（NotFound）-> 按本地策略处理
```

来源：[RFC 6811](https://datatracker.ietf.org/doc/html/rfc6811) 定义了 BGP 扩展团体属性用于传递验证状态。

## 4. BGPsec

BGPsec（[RFC 8205](https://datatracker.ietf.org/doc/html/rfc8205)）在 RPKI 基础上进一步保护 AS-PATH：

- 每条 BGPsec UPDATE 携带数字签名，证明路径中的每个 AS 确实收到了并转发了该前缀。
- 可防止 AS-PATH 篡改与部分路径伪造。
- 挑战：计算开销大、逐跳签名增加包大小、部署率低。

来源：[RFC 8205](https://datatracker.ietf.org/doc/html/rfc8205)、[RFC 8206](https://datatracker.ietf.org/doc/html/rfc8206)（BGPsec 设计选择）。

## 5. 部署现状与限制

| 方面 | 现状 | 限制 |
|---|---|---|
| RPKI/ROA | RIR 普遍支持，Cloudflare、Google、Microsoft 等积极部署 | 并非所有前缀都有 ROA；maxLength 配置错误会导致合法路由被误丢弃 |
| ROV | 大型 ISP 与 IXP 逐步启用 | 路由器 CPU/内存开销；策略保守导致未签名前缀仍被接受 |
| BGPsec | 实验与小范围部署 | 签名验证开销大；路径签名累积导致 UPDATE 尺寸膨胀 |

## 6. 工程实践

- **RPKI 依赖方软件**：Routinator、OctoRPKI、FORT、Cloudflare RPKI Validator。
- **路由器支持**：Cisco IOS XR、Juniper JunOS、Arista EOS、Nokia SR Linux 均支持 ROV。
- **策略模式**：
  - **ROV-enforcing**：丢弃 Invalid 起源。
  - **ROV-light**：仅记录或降低 Invalid 优先级，避免误伤未签名前缀。

## 7. 批判性分析

- **中心化风险**：RPKI 依赖 RIR/CA 层级，若 CA 私钥泄露或政策变化，可能影响全球路由。
- **maxLength 悖论**：过短的 maxLength 无法聚合，过长则降低安全性；运营商常配置不当。
- **BGPsec 的可扩展性**：逐跳签名对全网路由器性能构成挑战，实际部署进展缓慢。
- **激励不对称**：受害方承担劫持损失，而部署方承担成本，导致全球采用速度不均衡。
