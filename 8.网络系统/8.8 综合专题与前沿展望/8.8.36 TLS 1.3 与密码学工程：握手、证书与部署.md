## 1. 概述

TLS 1.3（[RFC 8446](https://datatracker.ietf.org/doc/html/rfc8446)）是 TLS 协议的重大更新，将典型握手压缩至 1-RTT，重连支持 0-RTT，并移除了大量过时密码套件。密码学工程不仅涉及协议本身，还涉及证书生命周期、PKI 治理、合规要求与部署细节。

## 2. TLS 1.3 握手流程

```text
客户端                                服务器
  | --- ClientHello + KeyShare + {pre_shared_key?} ---> |
  | <--- ServerHello + KeyShare + {EncryptedExtensions + Cert + CertVerify + Finished} --- |
  | --- {Finished + Application Data} ---> |
```

- **1-RTT**：首次握手在单次往返后发送应用数据。
- **0-RTT**：基于 pre-shared key 在第一个 RTT 中发送 early data，但存在重放风险，需应用层权衡。

完整握手示例包见 [RFC 8448](https://datatracker.ietf.org/doc/html/rfc8448)。

## 3. 密码学套件与密钥派生

TLS 1.3 仅保留 AEAD 套件，废除 RSA 密钥交换、CBC、RC4、MD5、SHA-1：

| 套件 | 密钥交换 | 认证 | 加密 |
|---|---|---|---|
| TLS_AES_256_GCM_SHA384 | (EC)DHE | RSA/ECDSA/EdDSA | AES-256-GCM |
| TLS_CHACHA20_POLY1305_SHA256 | (EC)DHE | RSA/ECDSA/EdDSA | ChaCha20-Poly1305 |
| TLS_AES_128_GCM_SHA256 | (EC)DHE | RSA/ECDSA/EdDSA | AES-128-GCM |

密钥派生使用 HKDF（Extract-then-Expand），从握手密钥派生流量密钥，实现前向保密。所有握手消息在 `Finished` 之后受到加密保护，提升了隐私性。

## 4. 证书与 PKI 工程

- **SNI（Server Name Indication）**：在 [RFC 6066](https://datatracker.ietf.org/doc/html/rfc6066) 中定义，允许客户端在握手早期告知目标主机名，使多租户 HTTPS 成为可能；TLS 1.3 通过 ECH（Encrypted ClientHello）进一步增强隐私。
- **max_fragment_length**：RFC 6066 允许受限设备协商更小的记录片大小。
- **证书管理**：CAB Forum Baseline Requirements 规定公开可信 CA 的密钥长度、有效期、域名验证流程；现代浏览器要求 TLS 服务器证书有效期不超过 398 天。
- **CT（Certificate Transparency）**：要求预证书与最终证书记录到公共日志，防止 CA 误发。
- **ACME 协议**：Let's Encrypt 使用 RFC 8555 自动化域名验证与证书签发，推动 HTTPS 普及。

## 5. 部署实践

- **服务器软件**：OpenSSL 1.1.1+、BoringSSL、NSS、GnuTLS、Microsoft SChannel、wolfSSL。
- **配置建议**：仅启用 TLS 1.3 与 TLS 1.2；优先 (EC)DHE 密钥交换；HSTS + OCSP Stapling；证书自动轮换（cert-manager、Let’s Encrypt）。
- **监控指标**：握手失败率、证书到期时间、支持的协议版本、密码套件分布。
- **客户端兼容性**：TLS 1.3 已被所有主流浏览器与移动 OS 支持，但部分旧版嵌入式设备仍需 TLS 1.2 回退。

## 6. 性能与可观测性

- **握手延迟**：TLS 1.3 1-RTT 握手比 TLS 1.2 的 2-RTT 显著降低；0-RTT 在重连场景接近无握手开销。
- **证书体积**：服务器可发送压缩证书（RFC 8879）以减少握手包大小。
- **可观测性**：加密握手导致被动网络监控困难；企业常依赖 TLS 检查代理或 ECHO 解密以进行安全审计。

## 7. 合规与审计要求

| 要求来源 | 关键要求 |
|---|---|
| NIST SP 800-52 Rev. 2 | 优先使用 TLS 1.3；禁用 SSLv2/v3、TLS 1.0/1.1；配置最小密钥长度 |
| PCI DSS 4.0 | 保护传输中的持卡人数据，使用强加密与最新协议版本 |
| CAB Forum BR | 公开 TLS 证书有效期不超过 398 天；域名验证需可审计 |
| 国密 SM 系列 | 中国部分场景要求 SM2/SM3/SM4 算法，需使用 GMTLS/TLCP 扩展 |

合规审计通常检查：证书链完整性、协议版本分布、密码套件强度、OCSP/CT 状态、私钥存储方式。

企业常将 TLS 配置纳入基础设施即代码（IaC），通过 Terraform、Ansible 或 Kubernetes ingress 注解实现统一基线，并定期使用 SSL Labs、testssl.sh 或 OpenSSL 命令行扫描暴露面。

## 8. 安全分析

| 风险 | 说明 | 缓解 |
|---|---|---|
| 0-RTT 重放 | early data 可能被攻击者重放 | 幂等接口、时间窗口限制、禁用 0-RTT |
| 降级攻击 | 中间设备迫使客户端使用 TLS 1.2 | 服务端禁用旧版本、支持 downgrade sentinel |
| 证书盗用 | 私钥泄露或 CA 被攻破 | 短有效期、CT、OCSP Must-Staple |
| 实现漏洞 | 如 OpenSSL 历史 Heartbleed | 及时补丁、最小化编译选项、FIPS 验证 |
| 侧信道 | 时序或缓存侧信道泄露私钥 | 恒定时间实现、HSM 保护 |

NIST SP 800-52 Rev. 2 为美国联邦机构提供 TLS 选择与配置指南。

## 9. 批判性分析

- **0-RTT 的安全-性能张力**：TLS 1.3 的 0-RTT 将延迟优化与安全保证之间的权衡显性化，需要应用层深度参与。
- **PKI 中心化风险**：全球 Web 信任锚集中于少数 CA 与浏览器/操作系统根证书计划，单点故障或地缘政治可能动摇信任基础。
- **后量子迁移压力**：现有 TLS 1.3 套件基于离散对数或椭圆曲线，面对通用量子计算机需引入抗量子密钥封装机制（如 ML-KEM），协议与生态将经历新一轮升级。
- **证书生态复杂度**：CT、OCSP、CRL、ACME、CT expectations 等多种机制并存，增加了运维与故障排查难度。
- **隐私与监控平衡**：TLS 1.3 加密了更多握手信息，企业安全团队需在用户隐私与威胁检测之间寻找新的合规边界，例如借助 ECH 与 MASQUE 等新技术重新设计可见性策略。
- **后量子迁移的兼容性**：在双证书或混合密钥交换方案成熟前，组织需同时维护传统算法与抗量子算法两套信任链，增加了部署复杂度。
