## 1. 概述

MQTT 与 CoAP 是物联网领域两类主流应用层协议。MQTT（Message Queuing Telemetry Transport）v5.0 由 OASIS 标准化，采用发布-订阅模型，适合高并发、低带宽、不稳定网络。CoAP（Constrained Application Protocol）由 IETF 定义，面向受限设备，借鉴 REST 模型，运行在 UDP 之上，支持组播与 observe 扩展。

## 2. MQTT v5.0 架构与机制

MQTT 使用发布-订阅模式，核心实体为发布者、代理（Broker）与订阅者：

```text
传感器 ──publish /topic/temp──→ Broker ←──subscribe /topic/temp── 应用
                                 ↓
                              消息分发（QoS 0/1/2）
```

MQTT v5.0 相对 v3.1.1 的关键增强：

| 特性 | 说明 |
|---|---|
| 原因码（Reason Codes） | 所有 ACK 包携带更细粒度的返回码 |
| 共享订阅（Shared Subscriptions） | 多个订阅者负载均衡接收同一主题消息 |
| 消息过期（Message Expiry） | 设置消息在 Broker 上的最大存活时间 |
| 主题别名（Topic Alias） | 减少重复主题字符串开销 |
| 用户属性（User Properties） | 可扩展的键值对头部 |
| 流控（Flow Control） | 接收最大报文数限制 |
| 增强认证（Enhanced Auth） | 支持挑战-响应式多步认证 |

QoS 等级：0（至多一次）、1（至少一次）、2（恰好一次）。

## 3. CoAP 架构与机制

CoAP（[RFC 7252](https://datatracker.ietf.org/doc/html/rfc7252)）基于 UDP，设计目标与 HTTP 类似但面向受限环境：

```text
客户端            服务器
  |--- CON GET /s/temp ---->|
  |<--- ACK 2.05 Content ---|
```

核心概念：

| 概念 | 说明 | 来源 |
|---|---|---|
| 消息类型 | CON（需确认）、NON（不需确认）、ACK、RST | RFC 7252 §4 |
| 请求/响应 | 方法码（GET/POST/PUT/DELETE）+ 响应码（2.xx/4.xx/5.xx） | RFC 7252 §5 |
| Option | URI-Path、Content-Format、Observe 等 | RFC 7252 §5.10 |
| Observe | 客户端订阅资源变化，类似长轮询 | RFC 7641 |
| Blockwise | 分块传输大 payload | RFC 7959（已更新至 RFC 8132） |
| CoAP over TCP/TLS | 适用于需要穿越 NAT/防火墙的场景 | RFC 8323 |

## 4. MQTT 与 CoAP 对比

| 维度 | MQTT v5.0 | CoAP |
|---|---|---|
| 传输层 | TCP / TLS / WebSocket | UDP / DTLS / TCP（RFC 8323） |
| 交互模式 | 发布-订阅 | 请求-响应 + Observe |
| 头部开销 | 2 字节固定头 + 变长编码 | 4 字节固定头 + Options |
| 可靠性 | QoS 0/1/2 | CON/ACK + 重传 |
| 适用网络 | 不稳定、低带宽、长连接 | 受限设备、 sleepy 节点、组播 |
| 安全 | TLS 1.3、用户名/密码、JWT | DTLS 1.3、EDHOC、OSCORE |
| 发现机制 | 依赖约定主题或外部注册 | RFC 6690 /.well-known/core |

## 5. 工程实现要点

- **MQTT Broker**：Eclipse Mosquitto、EMQ X、HiveMQ、VerneMQ、AWS IoT Core、Azure IoT Hub。
- **CoAP 实现**：Eclipse Californium、aiocoap、libcoap、Contiki-NG。
- **协议选择**：需要设备上线/离线状态、海量订阅时用 MQTT；需要 REST 接口、 sleepy 设备、组播时用 CoAP。
- **桥接模式**：网关同时运行 MQTT Broker 与 CoAP 服务器，完成协议转换。
- **资源模型**：CoAP 常与 LwM2M（OMA）结合，提供标准化的对象/资源管理。
- **测试与互操作**：使用 MQTT.fx、MQTT Explorer 或 mosquitto_sub/pub 验证 Broker；使用 Copper 插件或 libcoap 示例验证 CoAP 服务端。

## 6. 安全分析

- **MQTT 安全**：强制 TLS；避免默认匿名访问；细粒度 ACL；v5.0 增强的 AUTH 包支持增强认证。
- **CoAP 安全**：DTLS 1.3 提供传输安全；OSCORE（RFC 8613）提供端到端对象安全，适合多跳代理。
- **共同风险**：弱默认凭证、缺乏固件更新机制、Topic/URI 遍历、重放攻击。
- **证书管理**：大规模 IoT 部署常使用 X.509 证书或 PSK，需自动化证书生命周期管理。

## 7. 批判性分析

- **协议之争并非零和**：MQTT 更偏向中心化数据管道，CoAP 更偏向设备级 REST 接口；边缘网关中常同时存在。
- **标准化碎片化**：MQTT v5.0 属性丰富但实现一致性参差；CoAP 的 Observe、Blockwise、TCP 映射由多个 RFC 分别定义，增加集成复杂度。
- **长期维护挑战**：物联网设备生命周期长，协议版本升级与证书轮换困难，需在设计阶段预留 OTA 与密钥更新能力。
- **云厂商锁定风险**：主流云平台对 MQTT 扩展各异，跨云迁移时可能遇到认证、主题规则、QoS 行为差异。
- **规模与状态管理**：海量 MQTT 客户端同时在线时，Broker 的会话状态、保留消息与遗嘱消息管理成为瓶颈；CoAP 的无状态设计虽更易水平扩展，但 Observe 订阅状态仍需维护。
- **边缘与云协同**：MQTT Broker 常部署在边缘网关以减少云端往返；CoAP 则更适合设备直连本地网关，二者通过统一数据模型（如 WoT TD）实现语义互操作。
