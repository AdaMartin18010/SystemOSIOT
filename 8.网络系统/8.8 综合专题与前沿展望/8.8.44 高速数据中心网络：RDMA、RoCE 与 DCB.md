## 1. 概述与核心动机

现代 AI 训练、分布式存储与高性能计算（HPC）对网络提出低延迟、高吞吐、低 CPU 开销的要求。RDMA（Remote Direct Memory Access）允许一台主机直接读写远端主机的内存，绕过操作系统内核与 TCP/IP 协议栈，实现微秒级延迟。

| 指标 | TCP/IP 传统栈 | RDMA 网络 |
|---|---|---|
| 延迟 | 10–100 μs | <2 μs（InfiniBand）/ 2–5 μs（RoCEv2） |
| CPU 占用 | 高（内核协议处理、拷贝） | 低（硬件 offload、zero-copy） |
| 吞吐 | 受 socket 与中断限制 | 接近线速（100/200/400 Gbps） |
| 语义 | 字节流 | 单播/多播 SEND/RECEIVE、READ、WRITE、ATOMIC |

## 2. RDMA 语义与 InfiniBand 架构

IBTA（InfiniBand Trade Association）定义 InfiniBand Architecture，核心组件：

```text
应用 -> Verbs API -> RDMA 驱动 -> HCA（Host Channel Adapter）-> 链路 -> 交换机 -> 远端 HCA -> 远端内存
```

关键对象：

| 对象 | 说明 |
|---|---|
| QP（Queue Pair） | 发送队列与接收队列，是 RDMA 通信端点 |
| CQ（Completion Queue） | 完成事件队列，支持 poll 与 event 模式 |
| MR（Memory Region） | 注册到 HCA 的本地内存区域，含 L_Key/R_Key |
| SRQ（Shared Receive Queue） | 共享接收队列，减少内存占用 |
| WR（Work Request） | 下发给硬件的工作请求 |

RDMA 操作类型：

- **SEND/RECV**：双边操作，需远端预先 post receive。
- **READ**：远端读取本地内存，本地发起。
- **WRITE**：本地写入远端内存。
- **ATOMIC**：CAS/FAA 原子操作。

## 3. RoCEv1 与 RoCEv2

| 版本 | 网络层 | 特点 | 部署场景 |
|---|---|---|---|
| RoCEv1 | 以太网 L2 | 低延迟，但无法跨三层路由 | 单一 L2 域 |
| RoCEv2 | UDP/IP | 可路由，依赖 DCB/PFC 避免丢包 | 大规模数据中心 |

RoCEv2 封装：

```text
[Eth] [IP] [UDP dst=4791] [IB BTH] [Payload] [ICRC]
```

- **BTH（Base Transport Header）**：携带 QP、PSN、操作码等 InfiniBand 传输层信息。
- **ICRC**：Invariant CRC，覆盖 IP 不变字段与 IB 头部。

RoCEv2 依赖无损以太网（PFC）与拥塞通知（ECN/PFC/DCQCN）维持低延迟。

## 4. 数据中心桥接：PFC、ETS、DCBX、QCN

IEEE 802.1 数据中心桥接（DCB）标准族为 RoCE 提供无损与可预测转发：

| 标准 | 名称 | 功能 |
|---|---|---|
| IEEE 802.1Qbb | Priority-based Flow Control（PFC） | 按优先级 pause 上游，避免缓冲区溢出 |
| IEEE 802.1Qaz | Enhanced Transmission Selection（ETS）/ DCBX | 带宽分配与能力交换 |
| IEEE 802.1Qau | Quantized Congestion Notification（QCN） | 二层拥塞反馈 |

关键机制：

- **PFC**：将链路划分为 8 个优先级类，对 RoCE 流量单独 pause，不影响普通 TCP。
- **ETS**：为不同优先级分配最小带宽保证。
- **DCBX**：在链路两端交换 DCB 能力，自动配置 PFC/ETS。
- **QCN**：通过 CNM（Congestion Notification Message）反馈降低源端速率。

## 5. 拥塞控制与自适应路由

数据中心 RDMA 网络常见拥塞控制算法：

| 算法 | 机制 | 适用 |
|---|---|---|
| DCQCN | RoCEv2 基于 ECN 的端到端拥塞控制 | 广泛部署 |
| Timely | 基于 RTT 的速率控制 | Google 数据中心 |
| HPCC | 高精度拥塞控制，利用 INT 信息 | 超大规模 AI 训练 |
| ECMP | 等价多路径哈希 | 常规流量负载均衡 |
| Adaptive Routing | 交换机根据队列状态动态选路 | 减少incast热点 |

Incast 问题：多对一通信模式导致接收端交换机缓冲区瞬时拥塞，需结合 PFC、ECN 与合理的拥塞控制参数。

## 6. Linux RDMA 子系统与工程实现

Linux RDMA 子系统核心组件：

- **core**：设备管理、CM（Connection Manager）、SA（Subnet Administrator）代理。
- **mlx4/mlx5/rxe/irdma**：不同 HCA 驱动。
- **ipoib**：IP over InfiniBand。
- **srp/iser/nvme-of**：RDMA 存储协议。
- **libibverbs / librdmacm**：用户态 Verbs API 与连接管理库。

部署 checklist：

1. BIOS 启用 SR-IOV 与 Above 4G Decoding。
2. 安装 OFED / rdma-core 与对应内核模块。
3. 配置 PFC/ETS/DCBX，确保 RoCE 流量优先级一致。
4. 启用 ECN 并调优 DCQCN 参数。
5. 监控 RDMA 重传、PFC pause 次数、端口 CRC 错误。

## 7. 批判性分析

- **PFC 的副作用**： pause 帧可能向上游传播，引发 PFC storm 与全局 head-of-line 阻塞。
- **丢包敏感**：RoCEv2 依赖无损网络，单点配置错误即可导致大规模性能下降。
- **复杂度高**：DCB 配置涉及交换机、网卡、OS、应用多层级，排障困难。
- **成本**：InfiniBand 交换机与 NIC 成本高，RoCEv2 则需升级全栈以太网设备。
- **安全边界**：RDMA 绕过内核，传统防火墙与审计工具难以直接观测流量。
- **可扩展性**：超大规模集群中 ECMP 哈希不均与 incast 仍是持续优化方向。
