## 1. IPsec 架构概述

IPsec（Internet Protocol Security）在 IP 层提供机密性、完整性、数据源认证与抗重放保护。IPsec 架构由 [RFC 4301](https://datatracker.ietf.org/doc/html/rfc4301) 定义，包含两个核心协议：

- **AH（Authentication Header）**：[RFC 4302](https://datatracker.ietf.org/doc/html/rfc4302)，提供完整性认证与数据源认证，但不加密。
- **ESP（Encapsulating Security Payload）**：[RFC 4303](https://datatracker.ietf.org/doc/html/rfc4303)，提供加密、完整性认证与可选的抗重放。

IPsec 通过 **IKE（Internet Key Exchange）** 建立安全关联，IKEv2 规范为 [RFC 7296](https://datatracker.ietf.org/doc/html/rfc7296)。

## 2. 安全关联（SA）与 SPI

安全关联（Security Association, SA）是 IPsec 的核心状态：

```text
SA = { SPI, 目的 IP, 安全协议 (AH/ESP), 模式, 密码算法, 密钥, 生命周期 }
```

- **SPI（Security Parameters Index）**：32 位标识符，与目的 IP 和安全协议共同唯一确定 SA。
- SA 是单向的，双向通信需要一对 SA。

来源：[RFC 4301 §4.1](https://datatracker.ietf.org/doc/html/rfc4301#section-4.1)。

## 3. AH 与 ESP

| 协议 | 加密 | 完整性 | 抗重放 | 典型使用 |
|---|---|---|---|---|
| AH | ❌ | ✅ | ✅ | 仅需认证的场景（已较少使用） |
| ESP | ✅ | ✅ | ✅ | VPN、远程访问、站点互联 |

## 4. 传输模式与隧道模式

| 模式 | 封装方式 | 适用场景 |
|---|---|---|
| 传输模式 | 仅加密/认证 IP 载荷，保留原 IP 头 | 端到端主机通信 |
| 隧道模式 | 封装整个原 IP 包，添加新的 IP 头 | VPN 网关、站点到站点隧道 |

## 5. IKEv2 密钥交换

IKEv2 在 [RFC 7296](https://datatracker.ietf.org/doc/html/rfc7296) 中定义，使用两个交换阶段：

```text
阶段 1：IKE_SA_INIT
  协商加密/完整性/PRF/DH 组，建立 IKE SA

阶段 2：IKE_AUTH
  身份认证（证书、PSK、EAP），建立 CHILD_SA（即 IPsec SA）
```

IKEv2 支持：

- **完美前向保密（PFS）**：通过临时 DH 交换实现。
- **NAT 穿越**：[RFC 7296](https://datatracker.ietf.org/doc/html/rfc7296) 与 [RFC 3947](https://datatracker.ietf.org/doc/html/rfc3947) 定义 UDP 封装。
- **MOBIKE**：[RFC 4555](https://datatracker.ietf.org/doc/html/rfc4555) 支持 IP 地址变化时的连接迁移。

## 6. VPN 部署模式

| 模式 | 拓扑 | 典型协议栈 | 用例 |
|---|---|---|---|
| 站点到站点 | 网关 ↔ 网关 | IPsec 隧道模式 | 企业分支互联 |
| 远程访问 | 客户端 ↔ 网关 | IKEv2 + ESP | 移动办公 |
| 基于主机的端到端 | 主机 ↔ 主机 | IPsec 传输模式 | 高安全内网 |

## 7. 工程实现与 Linux 支持

- **Linux**：StrongSwan、libreswan、XFRM 框架、`ip xfrm`/`setkey`。
- **Windows/macOS/iOS/Android**：原生 IKEv2/IPsec VPN 客户端。
- **云网络**：AWS Site-to-Site VPN、Azure VPN Gateway、GCP Cloud VPN 均基于 IPsec/IKEv2。

## 8. 批判性分析

- **配置复杂性**：IPsec/IKEv2 的密码套件协商、证书链、策略匹配容易出错。
- **性能开销**：ESP 加密与 IKEv2 密钥交换增加 CPU 负载，高吞吐场景需硬件卸载。
- **中间设备兼容**：ESP 报文常被 NAT/防火墙阻断，需要 UDP 封装或端口转发。
- **与 TLS/QUIC 的分层竞争**：应用层加密（TLS 1.3、QUIC）已覆盖多数 Web 场景，IPsec 更适合网络层统一隧道。
