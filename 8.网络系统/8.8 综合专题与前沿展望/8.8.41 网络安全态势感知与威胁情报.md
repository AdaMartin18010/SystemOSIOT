## 1. 概述与核心目标

网络安全态势感知（CSA, Cyber Situation Awareness）通过持续收集、关联与分析多源安全数据，形成对网络威胁、资产暴露与防御效能的动态认知。威胁情报（TI, Threat Intelligence）则提供结构化、可机读的上下文信息，支撑检测、响应与预测。

核心目标：

- **可见性**：覆盖终端、网络、身份、云与供应链资产。
- **可理解性**：将原始告警转化为战术/运营/战略情报。
- **可行动性**：通过 SOAR、防火墙、EDR、SIEM 等自动或半自动响应。
- **共享性**：在安全社区、行业 ISAC 与公私伙伴间交换 IOC 与 TTP。

## 2. 威胁情报生命周期与共享模型

情报生命周期（参考 NIST SP 800-150）：

```text
规划与指导 -> 收集 -> 处理 -> 分析与生产 -> 传播与集成 -> 反馈与评估
```

| 层级 | 受众 | 典型内容 | 时效 |
|---|---|---|---|
| 战略 | CISO / 高管 | 攻击者动机、地缘政治风险、预算影响 | 季度/年度 |
| 运营 | SOC / IR 团队 | 战役时间线、目标行业、基础设施模式 | 周/月 |
| 战术 | 分析员 / 检测工程师 | TTP、IOC、YARA/Sigma 规则 | 小时/天 |
| 技术 | 自动防御系统 | IP、域名、哈希、URL、CVE | 分钟/小时 |

## 3. 态势感知数据融合与指标

态势感知通常聚合以下数据源：

| 数据源 | 内容 | 典型格式/协议 |
|---|---|---|
| 日志 | 认证、DNS、代理、VPN、云审计 | syslog、CEF、LEEF、JSON |
| 网络流量 | NetFlow/IPFIX、全包捕获、TLS JA3/JA4 | IPFIX、PCAP、Zeek |
| 终端遥测 | 进程、文件、注册表、API 调用 | EDR 原生格式、OpenC2 |
| 漏洞情报 | CVE、CVSS、CPE、KEV | NVD JSON、CVRF |
| 威胁情报 | IOC、STIX 2.1 对象 | STIX/TAXII、MISP JSON |

关键态势指标：

- **暴露面评分**：公网开放端口、脆弱资产占比。
- **检测覆盖率**：MITRE ATT&CK 技术点覆盖比例。
- **平均检测时间（MTTD）**与**平均响应时间（MTTR）**。
- **误报率 / 漏报率**：告警质量与分析师负荷。

## 4. MITRE ATT&CK 与攻击面映射

MITRE ATT&CK 是企业与移动威胁行为知识库，将攻击生命周期划分为战术（Tactics）与技术（Techniques）：

```text
初始访问 -> 执行 -> 持久化 -> 权限提升 -> 防御规避
   -> 凭据访问 -> 发现 -> 横向移动 -> 收集 -> 命令控制
   -> 数据窃取 -> 影响
```

工程应用：

- **检测缺口分析**：将现有 SIEM/EDR 规则映射到 ATT&CK 技术 ID，识别未覆盖技术。
- **红队/紫队演练**：基于 ATT&CK 设计模拟攻击路径并验证检测响应。
- **威胁狩猎**：针对特定 APT 组织的子技术（sub-technique）主动搜索痕迹。

## 5. STIX/TAXII、MISP 与自动化交换

| 标准/平台 | 作用 | 关键对象/能力 |
|---|---|---|
| STIX 2.1 | 结构化威胁信息表达 | Indicator、Observed Data、Threat Actor、Attack Pattern、Course of Action |
| TAXII 2.1 | 情报传输协议 | Collection、Channel、API（`/collections`、`/objects`） |
| MISP | 开源威胁情报共享平台 | 事件-属性-对象模型、分发规则、导出 STIX/TAXII、OpenIOC |
| VERIS | 事件分类与度量框架 | 行动、属性、影响、受害者向量 |

典型数据流：

```text
情报源（厂商、政府、开源）
        |
        v
STIX/TAXII 汇聚 -> MISP/威胁情报平台(TIP) -> SIEM/SOAR/防火墙自动阻断
```

NIST SP 800-150 强调共享前需进行**隐私与法律审查**、**分级标记**与**访问控制**，防止敏感信息外泄。

## 6. 工程实现与运营流程

- **情报源管理**：多源聚合，设置置信度与过期策略，避免 IOC 堆积。
- **上下文增强**：将 IP/域名关联到 ASN、GeoIP、被动 DNS、WHOIS 历史。
- **检测规则编排**：将 IOC 转换为 SIEM 查询、防火墙动态列表、DNS  Sinkhole。
- **响应自动化**：通过 SOAR 剧本实现告警分级、工单创建、隔离、取证。
- **度量与反馈**：定期复盘 MTTD/MTTR、情报命中率、规则误报。

## 7. 安全分析

| 风险 | 说明 | 缓解 |
|---|---|---|
| 情报误用 | 高置信度 IOC 被低质量情报源污染，导致误封 | 多源交叉验证、沙箱复核、白名单机制 |
| 共享合规 | 跨境共享可能违反 GDPR、网络安全法 | 数据分级、匿名化、法律审查 |
| 攻击者反制 | 威胁情报本身可被攻击者获取以规避检测 | 延迟分发、分发范围控制、诱饵 IOC |
| 工具孤岛 | SIEM、TIP、SOAR、EDR 未打通 | 统一数据模型、OpenC2/OpenTelemetry 标准接口 |
| 分析师过载 | 海量低质告警淹没关键事件 | 优先级评分、UEBA、自动关联 |

## 8. 批判性分析

- **情报时效与行动窗口矛盾**：高级持续威胁（APT）的 IOC 生命周期短，自动化响应若缺乏上下文可能误伤业务。
- **标准碎片化**：STIX/TAXII、OpenIOC、YARA、Sigma、CAPEC、MAEC 等并存，互操作成本仍高。
- **隐私与安全的张力**：为检测而广泛采集用户行为数据，可能侵犯隐私并扩大内部威胁面。
- **过度依赖指标驱动**：IOCs 易于过时，而基于 TTP 的行为检测需要更高分析能力但更具持久价值。
- **供应商锁定风险**：商业 TIP 与 SIEM 的专有数据格式可能削弱跨组织共享能力。
