## 1. 概述与演进背景

意图驱动网络（IBN, Intent-Based Networking）通过将业务意图转化为网络策略、配置与验证闭环，降低人工配置复杂度并提升一致性。网络数字孪生（NDT, Network Digital Twin）则在虚拟空间中构建物理网络的高保真模型，支持仿真、预测、验证与优化。

核心驱动力：

- **网络复杂度爆炸**：云原生、SD-WAN、5G 切片、SRv6 引入海量配置点。
- **业务敏捷需求**：DevOps 与 SRE 要求网络像软件一样可声明、可回滚。
- **AIOps 融合**： telemetry 数据驱动预测性运维与自动修复。
- **标准推动**：IETF YANG、NEMO、ETSI ZSM、MEF 3.0、ITU-T Y.3090 等。

```text
业务意图
    |
    v
意图转译器(IBN) -> 策略/配置 -> 网络控制器
    |                                |
    v                                v
数字孪生验证                物理网络执行
    |                                |
    v                                v
仿真/预测/优化 <-------------- Telemetry 反馈
```

## 2. 意图驱动网络（IBN）架构

IBN 典型参考架构（IETF/ONF 视角）：

| 层级 | 功能 | 示例 |
|---|---|---|
| 意图表达层 | 自然语言或结构化描述业务目标 | "视频会议 SLA：延迟 < 150ms，抖动 < 30ms" |
| 意图转译层 | 将意图映射为策略与配置 | 策略模型、约束求解、模板渲染 |
| 网络自动化层 | 下发配置并收集状态 | NETCONF/RESTCONF/gNMI、BGP-LS/PCEP |
| 验证与闭环层 | 持续验证意图满足度并自修复 | 形式化检查、仿真对比、AIOps 告警 |

IBN 与传统网络管理对比：

```text
传统: 人工CLI -> 逐设备配置 -> 事后排错
IBN : 声明意图 -> 自动转译 -> 持续验证 -> 自动修复
```

## 3. YANG 数据建模与标准化

YANG（Yet Another Next Generation）是 IETF 定义的数据建模语言，用于 NETCONF/RESTCONF 的管理数据：

- [RFC 6020](https://datatracker.ietf.org/doc/html/rfc6020)：YANG 1.0
- [RFC 7950](https://datatracker.ietf.org/doc/html/rfc7950)：YANG 1.1，支持 `action`、`notification`、`anydata` 等增强
- [RFC 8342](https://datatracker.ietf.org/doc/html/rfc8342)：Network Management Datastore Architecture（NMDA）

YANG 核心元素：

| 元素 | 说明 |
|---|---|
| container | 组织相关 leaf 的命名容器 |
| leaf / leaf-list | 单值/多值数据节点 |
| list | 可重复记录，含 key |
| rpc / action | 远程过程调用或数据节点动作 |
| notification | 事件通知模型 |

主流 YANG 模型：

- `ietf-interfaces`：接口配置与状态
- `ietf-routing`：路由配置
- `openconfig-*`：Google 等推动的 vendor-neutral 模型

## 4. NEMO 与意图表达

NEMO（NEtwork MOdeling，RFC 7120）提供面向网络的语言抽象，使非网络专家能够以业务语言描述网络需求：

```text
NEMO 语言示例概念：
Node(租户A) --[连接]--> Node(租户B)
Constraint: 带宽 >= 1Gbps, 延迟 <= 10ms
```

NEMO 与 YANG 的关系：NEMO 偏意图层抽象，YANG 偏数据模型层标准化；IBN 系统常将 NEMO 意图转译为 YANG 数据并下发。

## 5. 网络数字孪生（Network Digital Twin）

ITU-T Y.3090 将数字孪生定义为物理实体在数字空间的实时映射。网络数字孪生包含：

| 组件 | 说明 |
|---|---|
| 物理网络 | 真实设备、链路、流量、拓扑 |
| 虚拟映射 | 高保真拓扑模型、配置镜像、流量模型 |
| 数据连接 | telemetry、日志、配置变更同步 |
| 仿真/分析引擎 |  what-if 分析、故障注入、容量规划 |
| 应用接口 | 可视化、AIOps、策略验证 |

典型应用场景：变更前验证、故障定位与预测、安全演练、容量规划。

## 6. ETSI ZSM、MEF 3.0 与自动化闭环

| 框架 | 关注点 |
|---|---|
| ETSI ZSM（Zero touch network & Service Management）| 端到端零接触网络与服务管理，定义自动化闭环（闭环级别 L0-L4） |
| MEF 3.0 | 运营商级以太网与 SD-WAN 服务编排，强调意图驱动的服务生命周期管理 |
| ITU-T Y.3090 | 数字孪生网络框架与参考架构 |

ETSI ZSM 闭环级别：

- **L0 人工**：无自动化。
- **L1 辅助**：系统提供建议，人决策。
- **L2 部分自动化**：系统执行预定义任务，人监控。
- **L3 条件自动化**：系统在限定场景自主决策，人可接管。
- **L4 高度自动化**：系统在运营域内完全自主，人仅监督。
- **L5 完全自动化**：跨域端到端自治。

## 7. 工程实现要点

- **模型一致性**：物理网络与数字孪生需保持配置、拓扑、状态同步。
- **YANG 模型选择**：在 IETF 标准模型、OpenConfig 与厂商私有模型间做兼容封装。
- **意图转译鲁棒性**：意图可能存在歧义，需结合约束求解与人工确认。
- **数据规模**：telemetry 高频采样带来海量时序数据，需分布式存储与流处理。
- **安全与信任**：控制器与孪生平台成为关键基础设施，需零信任访问与变更审计。

## 8. 批判性分析

- **意图歧义与转译风险**：自然语言意图可能隐含多种解释，错误转译会导致全局配置错误。
- **模型覆盖不足**：YANG/OpenConfig 尚未覆盖所有厂商特性，异构网络难以完全统一建模。
- **孪生保真度与成本**：高保真孪生需要大量数据与算力，中小组织难以承受。
- **闭环自动化的责任边界**：L4/L5 自动化下，故障责任归属与安全回滚机制需明确。
- **供应商锁定**：IBN/NDT 平台深度绑定控制器与数据模型，迁移成本高。
- **“自智网络”营销泛化**：部分产品仅将简单脚本包装为 IBN，缺乏真正的意图验证与闭环。
