## 1. 概述

TCP 与 UDP 长期主导互联网传输层，但特定场景催生了多种增强或替代方案：TCP Fast Open（TFO）降低握手延迟；BBR 优化高带宽时延网络；Multipath TCP（MPTCP）聚合多条路径；SCTP 提供消息边界与多宿；DCCP 面向需要拥塞控制但不可靠传输的媒体流。它们共同反映了互联网从“尽力而为”向“场景定制化”演进的趋势。

## 2. TCP Fast Open

[TCP Fast Open, RFC 7413](https://datatracker.ietf.org/doc/html/rfc7413) 允许在 SYN 包中携带数据，将常见 Web 请求的 1-RTT 进一步缩短：

```text
传统 TCP + TLS 1.3        TCP Fast Open + TLS 1.3
  SYN  →                     SYN + TFO Cookie + data  →
  SYN-ACK ←                  SYN-ACK + data + TLS  ←
  ACK + TLS ClientHello →    ACK + 应用数据  →
```

- 服务器通过 `TFO Cookie` 验证客户端身份，降低 SYN 洪泛风险。
- Cookie 通常由服务器 IP 与密钥派生，具有一定有效期。
- Linux 内核自 3.7 起支持客户端，3.11 起支持服务端；Windows Server 2016+ 支持。

## 3. BBR 拥塞控制

BBR（Bottleneck Bandwidth and Round-trip propagation time）由 Google 提出，与基于丢包的 CUBIC（[RFC 8312](https://datatracker.ietf.org/doc/html/rfc8312)）不同，它基于模型驱动：

| 参数 | 含义 | 测量方式 |
|---|---|---|
| BtlBw | 瓶颈带宽 | 发送-确认速率的最大值 |
| RTprop | 传播往返时延 | 最小 RTT 观测值 |
| BDP | 带宽时延积 | BtlBw × RTprop |

BBR 通过 pacing 发送、避免过度填充队列，在高 BDP 网络（卫星、5G、跨洲链路）表现优于传统 Reno/CUBIC。BBR v2 进一步改善了与 CUBIC 的公平性并引入 ECN 响应。

## 4. Multipath TCP

MPTCP（[RFC 8684](https://datatracker.ietf.org/doc/html/rfc8684)）在 TCP 之上抽象一条逻辑连接，可绑定多个子流：

```text
MPTCP 连接
 ├── Subflow 1 : Wi-Fi (192.168.x.x)
 └── Subflow 2 : 5G (10.x.x.x)
```

- 支持包级调度、路径管理与无缝切换。
- 苹果自 iOS 7 起在 Siri 中使用 MPTCP；Linux 内核有 MPTCP 子系统。
- 路径管理策略包括全mesh、主备、增量等，调度器可按 RTT、拥塞窗口或带宽选择路径。

## 5. SCTP 与 DCCP

| 协议 | 特点 | 典型场景 | 来源 |
|---|---|---|---|
| SCTP | 消息边界、多宿主、多流、INIT 防 SYN 洪泛 | SIGTRAN、WebRTC 数据通道（旧版） | RFC 4960 |
| DCCP | 不可靠但拥塞受控的数据报 | 媒体流、在线游戏 | RFC 4340 |

SCTP 的多宿主能力允许一个端点拥有多个 IP 地址，在单条路径失效时自动切换。DCCP 提供 CCID（拥塞控制标识）协商，允许应用选择适合的拥塞控制算法。

SCTP 的受限穿透 NAT/防火墙能力限制了其在公网 Web 的普及，DCCP 则因生态支持不足而较少部署。

## 6. 对比与工程选择

| 场景 | 推荐方案 | 理由 |
|---|---|---|
| Web 低延迟 | TFO + TLS 1.3 / QUIC | 减少握手 RTT |
| 高 BDP 网络 | BBR | 低缓冲区占用、高吞吐 |
| 移动/多宿主 | MPTCP | 链路聚合与无缝切换 |
| 电信信令 | SCTP | 消息边界与高可用 |
| 实时媒体 | DCCP / RTP over UDP | 容忍丢包，避免 TCP 重传延迟 |

## 7. 操作系统与中间件支持

- **Linux**：TFO 通过 `/proc/sys/net/ipv4/tcp_fastopen` 启用；BBR 作为 tcp_congestion_control 模块；MPTCP 子系统自 5.6 起合并主线。
- **Windows**：TFO 在 Server 2016+ 可用；BBR 未原生支持；SCTP 通过第三方驱动或用户库。
- **中间件**：Envoy、nginx 支持 TFO；Cloudflare 大规模部署 BBR；HAProxy 实验性支持 MPTCP。
- **调试工具**：`ss -ti` 可查看 TCP 拥塞控制算法与 BBR 状态；`mptcpize` 可将传统应用透明转换为 MPTCP。

## 8. 安全分析

- **TFO Cookie 重放**：攻击者可能重放携带数据的 SYN，应用层需保证幂等性。
- **MPTCP 路径劫持**：新增子流需通过密钥认证（MP_JOIN），但仍需依赖 TLS 提供端到端安全。
- **SCTP INIT 放大**：需限制 INIT-ACK 中地址块数量与状态 Cookie 大小。
- **BBR 公平性**：在 BBR 与 CUBIC 共存网络中，BBR 可能占用更多瓶颈带宽，引发公平性争议。

## 9. 批判性分析

- **部署碎片化**：TFO/BBR 受 OS 与中间件限制；MPTCP/SCTP/DCCP 在公网穿透困难。
- **QUIC 的替代效应**：QUIC 将连接建立、多路复用、拥塞控制合入用户空间，部分取代 MPTCP 与 SCTP 的设计目标。
- **标准化与产业落地差距**：SCTP 与 DCCP 标准化较早，但浏览器与中间件支持不足，形成“标准存在、生态缺失”的局面。
- **拥塞控制公平性**：不同算法共存时，基于模型的 BBR 与基于丢包的 CUBIC 在缓冲区有限的瓶颈处行为差异显著，需网络管理员审慎调参。
- **测量与诊断难度**：TFO 与 0-RTT 类优化将数据提前到握手阶段，传统抓包与防火墙规则可能误判为异常流量。
- **与 QUIC 的共存**：在 QUIC 普及区域，部分 MPTCP 与 SCTP 用例被 HTTP/3 的流多路复用替代，但在内核层与电信核心网仍有不可替代场景。
