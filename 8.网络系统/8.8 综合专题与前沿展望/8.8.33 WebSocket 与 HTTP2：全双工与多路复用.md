## 1. 概述

WebSocket（[RFC 6455](https://datatracker.ietf.org/doc/html/rfc6455)）与 HTTP/2（[RFC 9113](https://datatracker.ietf.org/doc/html/rfc9113) 取代 RFC 7540）是现代 Web 与应用通信的两大关键协议。HTTP/2 通过二进制分帧与多路复用提升传统请求-响应模型的效率；WebSocket 则在单条 TCP 连接上提供全双工、低延迟的消息通道，弥补 HTTP 半双工的局限。HTTP 语义由 [RFC 9110](https://datatracker.ietf.org/doc/html/rfc9110) 与 [RFC 9112](https://datatracker.ietf.org/doc/html/rfc9112) 统一规范。

## 2. WebSocket 协议机制

WebSocket 握手基于 HTTP Upgrade：

```text
客户端                服务器
  |--- GET /chat HTTP/1.1 + Upgrade: websocket --->|
  |<-- 101 Switching Protocols + Sec-WebSocket-Accept --|
  |<================= 全双工帧通道 =================>|
```

关键字段：

| 字段 | 作用 | 来源 |
|---|---|---|
| `Sec-WebSocket-Key` | 客户端 16 字节随机数 Base64 | RFC 6455 §4 |
| `Sec-WebSocket-Accept` | 服务器对 Key 做 SHA-1 + Base64 | RFC 6455 §4 |
| `Sec-WebSocket-Protocol` | 协商子协议（如 MQTT/CoAP） | RFC 6455 §4 |
| `Sec-WebSocket-Extensions` | 协商扩展（如 permessage-deflate） | RFC 6455 §9 |

帧格式包括 FIN、opcode（text/binary/close/ping/pong）、MASK（客户端必置位）、Payload len 与扩展长度。掩码旨在防止缓存污染攻击，要求客户端到服务器的数据必须异或掩码。

## 3. HTTP/2 帧层与多路复用

HTTP/2 将请求/响应拆分为二进制帧：

```text
HTTP/2 连接
 ├── Stream 1 :HEADERS :DATA :DATA (请求/响应 A)
 ├── Stream 3 :HEADERS :DATA (请求/响应 B)
 └── Stream 5 :HEADERS :DATA :RST_STREAM
```

核心机制：

| 机制 | 说明 | 来源 |
|---|---|---|
| 多路复用（Multiplexing） | 多条流共享一条 TCP 连接，消除应用层队头阻塞 | RFC 9113 §5 |
| 头部压缩 HPACK | 静态表、动态表与哈夫曼编码 | RFC 7541 |
| 流优先级 | 依赖与权重控制资源调度 | RFC 9113 §5.3 |
| 服务器推送 | 服务器主动发起响应（现代浏览器多已禁用） | RFC 9113 §8.2 |
| 流量控制 | 基于 WINDOW_UPDATE 的连接级与流级流控 | RFC 9113 §5.2 |

HTTP/2 仍受 TCP 层队头阻塞影响，这正是 QUIC/HTTP/3 的设计动机之一。

## 4. WebSocket 与 HTTP/2 的协同

传统 WebSocket 基于 HTTP/1.1 Upgrade。RFC 8441 定义了在 HTTP/2 之上建立 WebSocket 连接的扩展，利用 HTTP/2 的 `CONNECT` 方法携带 `:protocol` 伪头：

```text
:method = CONNECT
:protocol = websocket
:authority = example.com
:path = /chat
:scheme = https
```

该机制允许 WebSocket 与 HTTP/2 共享同一条 TLS 连接，减少连接数；但浏览器与中间件支持仍不如 HTTP/1.1 Upgrade 广泛。

## 5. 两者对比

| 维度 | WebSocket | HTTP/2 |
|---|---|---|
| 通信模式 | 全双工消息 | 半双工请求-响应 |
| 连接建立 | HTTP Upgrade → 101 | TLS/ALPN（h2）或明文升级 |
| 适用场景 | 实时推送、游戏、聊天 | REST API、网页加载、视频 |
| 多路复用 | 无（单连接单流） | 有（多流共享连接） |
| 头部开销 | 帧头 2–14 字节 | 头部经 HPACK 压缩 |
| 传输层 | TCP / TLS | TCP / TLS |

实践中二者可互补：HTTP/2 负责静态资源与 API，WebSocket 负责实时双向通道；在 QUIC 时代，WebTransport 尝试在 Web 层提供类 WebSocket 的低延迟流能力。

## 6. 工程实现要点

- **服务器框架**：Node.js `ws`、Spring WebFlux、Go `gorilla/websocket`、uWebSockets。
- **HTTP/2 服务器**：nginx、h2o、Apache、Envoy、Caddy。
- **负载均衡**：HTTP/2 需 L7 代理维护流状态；WebSocket 需支持 Upgrade 与长连接，Layer 4 负载均衡更为常见。
- **心跳与超时**：WebSocket 推荐应用层 ping/pong 或协议级 ping/pong 帧；HTTP/2 使用 `SETTINGS` 与 keep-alive。
- **压缩**：WebSocket 可协商 permessage-deflate；HTTP/2 不允许压缩头部之外的 body 强制压缩，需 Content-Encoding 协商。

## 7. 安全分析

- **WebSocket 安全问题**：缺乏同源策略强制，需 CSRF 防护；wss:// 必须用于生产；握手可能被绕过以建立非预期通道；消息掩码不解决端到端加密问题。
- **HTTP/2 安全问题**：HPACK 炸弹、Stream 0 RST 放大、依赖长度前缀的攻击面；需限制并发流数、帧大小与头块大小。
- **TLS 必要性**：二者在公网均应经 TLS 1.3 保护，HTTP/2 通常依赖 ALPN 协商 `h2`。

## 8. 批判性分析

- **协议竞争与冗余**：WebSocket 的低延迟消息与 HTTP/2 的服务器推送部分重叠，但后者因代理与缓存问题被边缘化。
- **TCP 队头阻塞**：HTTP/2 的多路复用无法避免 TCP 层重传阻塞，催生了基于 UDP 的 HTTP/3。
- **可观测性下降**：HTTP/2 二进制帧与 WebSocket 帧加密化（wss）后，传统基于文本的调试与审计工具受限。
- **HTTP/2 over WebSocket 的部署困境**：RFC 8441 理论上优雅，但现有基础设施与浏览器实现推进缓慢。
