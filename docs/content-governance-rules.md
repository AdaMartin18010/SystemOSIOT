# SystemOSIOT 内容治理规则（Phase 0 基线）

> 生效日期：2026-07-02
> 适用范围：全项目所有 Markdown 文件与目录
> 治理目标：遏制过度递归、统一国际标准对齐方式、保证形式化工件可追踪。

## 1. 目录深度限制

- **最大递归深度 ≤ 5**：任何新增子目录或文件的层级编号不得超过 5 级（如 `7.1.2.3.4` 为最大允许深度）。
- 现有 6 级及以上文件进入 **清理队列**，由维护者合并、迁移或删除。
- 例外：仅 `docs/`、`tools/`、`validation/`、`research/` 等元目录可不受此限制，但应避免无意义嵌套。

## 2. 文件命名规范

- 文件名禁止重复出现“子主题递归细化”等无信息冗余词汇。
- 文件名应使用 **编号 + 英文/中文核心主题词**，例如：
  - ✅ `7.1.6.2.1 AI微服务多模型路由.md`
  - ❌ `7.1.6.2.1.5.1.1.1.1 AI微服务多模型管理与自适应路由子主题递归细化子主题.md`
- 同一主题下禁止创建标题高度相似（Jaccard 相似度 > 0.8）的多个文件。

## 3. 国际标准对齐要求

- 每个主题模块必须包含 `X.0 国际标准映射/` 子目录。
- 引用任何标准、论文、协议、产品时，必须标注：
  - 标准/论文/协议全称
  - 版本号或年份
  - 官方链接或 DOI
  - 访问日期
- 优先引用 **ISO / IEC / IEEE / IETF RFC / ACM / USENIX / 3GPP** 等权威来源，其次为官方文档与经典教材。

## 4. 形式化工件要求

- 任何声称“已形式化证明/验证”的命题，必须关联至少一个可执行或可检查的工件：
  - Coq (`.v`)
  - Isabelle/HOL (`.thy`)
  - Lean 4 (`.lean`)
  - TLA+ (`.tla` / `.cfg`)
  - 模型检验器输入（`.smv`、UPPAAL XML、PRISM `.prism`、Alloy `.als`、SMT-LIB `.smt2`）
- 禁止仅通过 Markdown 文字声明“已形式化验证”而不提供可运行工件。

## 5. 内容创建流程

1. 新增文件前检查是否已存在同一主题的父级文件；优先在父级补充内容，而非创建新子文件。
2. 新增递归子文件需经过维护者评审，理由应写入文件头部的 `<!-- 创建理由：... -->` 注释。
3. 每季度执行一次目录深度扫描与重复标题检测。

## 6. 违规处理

- 对违反上述规则的新增文件，CI 的 TOC 校验门增加目录深度检查与标题相似度检查。
- 历史违规文件进入 `docs/refactor/cleanup-queue.md` 清理队列，分批处理。
