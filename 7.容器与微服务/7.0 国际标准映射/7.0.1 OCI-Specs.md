### 2.2 运行时实现对照

| 实现 | 对应层级 | 说明 |
|---|---|---|
| runC | 低层 OCI 运行时 | 默认参考实现，直接操作 Linux 命名空间与 cgroups |
| crun | 低层 OCI 运行时 | C 语言实现，性能更高，支持 cgroups v2 |
| containerd | 高层运行时（CRI 实现） | 调用 runC/crun，管理镜像、网络、存储 |
| CRI-O | 高层运行时（CRI 实现） | 专为 Kubernetes 设计，轻量 |
| Kata Containers | VM-based 安全运行时 | 使用 QEMU/Cloud Hypervisor 提供 VM 级隔离 |
| gVisor | 用户态内核 | 拦截系统调用，提供额外隔离 |

### 3.2 镜像供应链安全

OCI v1.1 引入 `referrers` 后，镜像签名（cosign/sigstore）、SBOM（Syft、Trivy）、漏洞报告均可通过 referrers API 关联。项目应在 `7.1 知识梳理` 中增加：

- 镜像签名与验证（Sigstore/cosign、Notation）
- SBOM 与 provenance
- Referrers API 查询示例

## 5. 与 Kubernetes 的关系

Kubelet 通过 CRI 调用 containerd/CRI-O，后者通过 OCI Runtime 启动容器，通过 OCI Image 拉取镜像，通过 OCI Distribution 与镜像仓库交互。项目应在 `7.3 形式化结构` 中明确这一分层：

```text
Kubernetes API
    ↓ CRI (gRPC)
containerd / CRI-O
    ↓ OCI Runtime Spec    runC / crun / Kata / gVisor
    ↓ OCI Image Spec      manifest / config / layers
    ↓ OCI Distribution    Registry (Harbor / Docker Hub)
```
